# AI Agent 的知行合一——从淀粉贸易系统看智能体的本质

---

## 引言

这是一个用飞书多维表格替代传统ERP的贸易管理系统。13张表、60+公式字段、200+条业务记录，通过自然语言对话完成采购、销售、回款、物流、财务的全链路操作。

但本文不是关于"怎么做业务管理"的。本文是关于一个AI Agent如何在真实、混乱、不完美的环境中，理解业务意图、执行复杂规则、在失败中自校正、最终成长为"有经验的同事"的完整记录。

---

## 一、五维能力栈：一个Agent从"能执行"到"可信任"

Agent的能力不是一维的，不是"能调API"就完事了。它是一层层垒上去的——每一层都依赖于下一层，每一层都在解决"上一层的不可靠"。

### 第一层：感知——在非结构化的自然语言中发现结构

这一层要回答的问题不是"用户的提示词我能不能看懂"，而是"用户的提示词和我预设的模板之间有多大的差异，我能容忍多大的差异"。

在这个项目中，DeepSeek承担了感知层的核心任务。用户日常输入的可能是口语化的、格式随意的表述——"给王奎发32吨绿五叶，4200一吨，仓库发货"。这不需要是完美的、不需要包含所有字段、不需要遵循严格格式。DeepSeek的职责是将这种不规整的自然语言，按照预设好的提示词模板，**翻译**成Hanako能精确解析的标准化格式：

```
【销售送货🚛_仓库发货📦：送给【王奎👥】约定单价为 4,200元💵每吨的【🍀绿五叶📦】的销售订单 32吨⚖️，送货方式是【仓库发货📦】，关联客户、商品和对应订单。】
```

这里的关键设计理念是：**模板不是给Agent看的，是给Agent间的通信设计的**。模板保证了DeepSeek的输出和Hanako的输入在同一个语义空间内。`【...】`是原子记录边界，`👥📦💵⚖️`是类型标记——这不是装饰性的emoji，而是一种**领域特定语言（DSL）的类型系统**。客户标记锁定实体类型为"人"，商品标记锁定实体类型为"物"，吨数标记锁定数值单位为"物理重量"，价格标记锁定数值单位为"货币"。Agent用这些标记解析字段，而不是靠语义猜测。

这与传统NLP的本质区别在于：传统方式试图理解"王奎"是一个人名、"绿五叶"是一个产品名——这是对开放世界的语义理解，准确率有天花板。而这里的标记系统创造了**一个闭合的、协商好的微型语言**——在这个语言里，格式本身就是类型保证。

### 第二层：记忆——从"全忘干净"到"越用越好"

Agent有四种不同的"遗忘"方式，每一种都需要不同的对策：

**第一种：对话Context Window截断。** 一个长时间的操作会话可能涉及几十次API调用、上百条记录的创建和验证。当对话历史超出窗口限制时，Agent会忘记前面做过什么、查过哪些档案ID、匹配过哪些订单。解决方案是Session内的State管理——在同一个evaluate上下文内完成尽可能多的操作，避免跨turn的状态丢失。

**第二种：跨Session的经验丢失。** 上一个会话踩过的坑，下一个会话又踩一遍。DuplexLink空壳陷阱、浮点精度匹配失败、反向链接覆盖——这些错误在不同Session间反复出现。解决方案是Skill文件——一个独立于对话Context Window的持久化规则库。12条铁律每一条都是从具体的失败中提炼出来的。Skill不是代码——它是Agent的"长期记忆"，每次触发时从文件加载，每次失败后写入更新。这就形成了一个**持续学习的闭环**——不是训练模型的权重，而是优化Agent的运行时约束。

**第三种：表格数据的同步延迟。** 一次写入后立即回读，但同事同时在操作同一张表格，数据已经变了。解决方案是实时GET查询——不缓存档案ID，不依赖之前查询的结果，每次操作前重新拉取。

**第四种：沉默的错误。** code=0不代表链接成功。DuplexLink返回码是0但record_ids为空——这是一个"看起来对了但实际错了"的情况。解决方案是Fehler-proofing（防误设计）：每条记录创建后逐条回读验证关联字段的record_ids非空。不是"相信上一步没问题"，而是"假设上一步可能有问题"。

### 第三层：规划——不是"剧本"是"决策树"

传统自动化脚本的执行路径是线性的：step1→step2→step3，任何一步异常就终止。Agent的规划不是这样——它是条件分支的，每一步的输出决定下一步走哪条路径。

本系统的规划核心是一个两层的决策节点：

**节点A：Token获取。** 尝试app token → 成功则继续；返回404则意味着IP被封 → 自动切到OAuth用户token路径。这不是两个独立的"脚本"，而是一个决策——Agent根据环境反馈选择正确的路径。

**节点B：回款超额处理。** 正常创建回款 → 检测到回款率将超过100% → 触发FIFO拆分分支 → 删旧建新，逐单分配。这条路径不是预设的"正常流程"，而是Agent在"看到"超额风险后自主触发的一个备用逻辑。

这种规划的本质是：**Agent不是在执行一个预设脚本，而是在根据每一步的输出持续做选择**。这就像下棋——你不可能预写一个"必胜棋谱"，只能根据对手的每一步调整自己的下一步。

### 第四层：工具使用——"知道工具缺陷"比"会用工具"更重要

API文档里不会写的内容：DuplexLink可以创建空壳、PUT是替换不是追加、公式内容不通过API暴露、浏览器有两个出口IP需要分别白名单、open.feishu.cn首页有CSP限制、OAuth码是一次性的。

这些不是"功能特性"，是"系统缺陷"。但正是在这些缺陷上，区分了"能调API的脚本"和"真正的Agent"。

本系统对工具使用有三个层次的理解：

**第一层——正确的格式：** DuplexLink写入必须用纯数组`["record_id"]`而非对象格式，响应解析必须用`text()+JSON.parse()`而非`.json()`。这是最低要求。

**第二层——后验证：** code=0不代表链接生效。每条记录创建后逐条GET回读，检查record_ids非空。不通过就PUT重写，再回读。

**第三层——缺陷认知与路径切换：** 当某条路径不通（app token被IP拦截），自动识别失败原因（404=IP问题），切换到备用路径（OAuth用户token）。这不是"换了种调用方式"，而是Agent**理解了工具限制**后的自主路由。

### 第五层：自主决策——理解"业务语义"的真正含义

很多系统能执行"创建一条回款记录"——这就是"能调API"的水平。但真正的自主决策在于理解**这条回款应该分配给哪个订单、分配多少、会不会超额、会不会绑错人**。

FIFO回款拆分展现了自主决策的完整链条：

```
输入：一笔回款，已知客户身份
判断1：该客户有多少未结清订单？按时间排序
判断2：第一个订单的待回金额是多少？≤本次回款？
判断3：如果≤，则全部分配给此订单，剩余金额继续分配给下一个
判断4：新创建的每条回款，只关联一个订单
判断5：运行验证——所有订单回款率≤100%，所有关联客户名一致
```

这不是"执行指令"——这是**理解业务规则后的自主分配**。Agent不是在"创建一条回款记录"，而是在"把一笔回款按照先进先出原则，智能分配到多个未结清订单中，并确保不超额、不跨客户"。

## 二、多Agent协作：流水线而非堆叠

三个Agent构成了一个三阶段的处理流水线，每个Agent解决一个不同的可靠性问题：

**第一棒：DeepSeek——模板化。** 将用户不规整的自然语言转成标准化提示词。这解决的是"输入不可靠"的问题——用户想表达的和系统能理解的之间存在鸿沟。DeepSeek的任务是架桥：它理解用户的意图（"给这个人发这个产品"），然后填入预设的模板生成标准格式。这保证了进入系统的每条指令都是可精确解析的。

**第二棒：扣子(Coze)——定时与调度。** 在指定时间触发经营分析，生成报告推送给用户。这解决的是"操作不可靠"的问题——人不可能每天定时做分析、手工汇总。扣子是"不需要人记住的时间触发器"。

**第三棒：Hanako——执行与验证。** 接收标准化提示词，解析、查档案、创建记录、关联验证、报告结果。这解决的是"执行不可靠"的问题——即便是标准化的指令，在真实的API环境和数据约束下，执行也可能出错。Hanako的价值不在于"调用API"，而在于"在API出错时知道怎么回事、能自己修、修完还能标记防止下次再犯"。

三个Agent之间的通信协议不是MCP或JSON RPC——而是**自然语言提示词 + 结构化回复报告**。这种设计的选择基于一个核心认知：当Agent共享同一套emoji标记体系和业务语义时，自然语言本身就是最有效的通信方式。不需要引入额外的协议层——协议已经内隐在格式里了。

## 三、自然语言的"结构化"——不靠AI猜，靠格式保证

一个常见误解是：自然语言交互意味着AI去"理解"任意表达方式。但这个项目表明，真正可靠的自然语言系统不是靠AI的语义理解能力，而是靠**设计一套AI和人都容易遵守的格式化语言**。

这套标记体系的设计原则：

- `【...】` = 一条原子业务记录的边界。任何在这个边界内的内容，属于同一条记录
- `👥👤🏭` = 实体类型标记。告诉Agent接下来是一个客户/供应商的名称
- `⚖️💵💰` = 数值类型标记。告诉Agent接下来是一个重量/销售价格/采购价格
- `📦🚛💳` = 操作类型标记。告诉Agent这个送货方式是仓库发货/车上直发/信用卡

这不是为了让提示词好看。这是为了让Agent**不需要做类型推断**。当Agent看到`【王奎👥】`时，它知道"王奎"是一个客户名，不是供应商名、不是商品名、不是一个备注文本——因为`👥`标记了它的类型。这消除了传统NLP中的实体歧义问题——同样的文字"李伟"，在客户表可能是客户名，在供应商表可能是供应商名，但在有类型标记的系统中，歧义在**标记层**就已经解决了。

## 四、防御性设计：Agent的"免疫系统"

生产级的Agent与概念验证的Agent之间的差距，不在"正常情况下的功能"，而在"异常情况下的表现"。

### 4.1 五层校验的防误设计

第一层——API接受检查。code=0，请求被API接受了。最低限度的通过信号，但远不够。

第二层——结构完整性检查。创建的记录确实存在于表中，GET回读能拿到record_id。

第三层——关联有效性检查。这条送货记录声称关联了订单X，那么GET这条记录的对应订单字段，record_ids非空吗？不为空就进入下一层。

第四层——关联正确性检查。关联的订单X确实是这个客户的订单吗？客户名交叉校验——这是"关联不跨客户"的保证。

第五层——业务规则检查。回款率不超过100%，发货率不超过100%，客户无重名。

每一层都假设上一层的结果可能是错的。不是"相信API返回了成功"，而是"检查API返回的到底是不是真的成功"。

### 4.2 自我免疫的进化机制

每发现一个错误，不仅修复当前数据，还提取一条通用规则写入Skill。这创建了一个**自适应系统**——它会随着使用次数增加而变得越来越可靠。使用的次数越多，遇到的边界情况越多，免疫系统就越完善。

这不是机器学习中的fine-tuning，而是**运行时约束的持续优化**——Agent的学习不是在训练数据集上完成的，而是在每一次真实操作中完成的。

## 五、系统的亮点：从"表格工具"到"业务操作系统"

### 亮点一：零代码的ERP

不需要安装任何软件、不需要部署服务器、不需要写SQL。飞书多维表格的公式引擎（60+跨表聚合公式）替代了传统ERP中最复杂的报表逻辑。财务汇总的8项指标——现金、应收、库存、在途、应付、贷款、已售、净资产——全部通过公式字段实时聚合。任何一条业务记录的变更，都会在秒级传导到汇总层。

### 亮点二：自然语言即操作界面

从"打开表格→找到对应行→填入数据→选择下拉框→保存"到"说一句话"。不仅效率提升，更降低了使用门槛——不需要理解表结构、字段关系、关联逻辑，这些都内化在Agent的知识里。

### 亮点三：越用越好的自学习系统

12条铁律不是一次性写成的，而是在40+次失败中一条条长出来的。Skill文件是Agent的"长期记忆"，每次失败后的修正不是"今天修好了"，而是"从今以后都不会再犯"。

### 亮点四：数据完整性的防御体系

五层校验、双向链接、客户名交叉验证、FIFO回款拆分——这些不是锦上添花的功能，而是保证"数据不出错"的基础设施。在多Agent协作、多用户操作的环境下，数据完整性是第一优先级。

### 亮点五：可迁移的多Agent架构

DeepSeek（模板化）+ 扣子（调度）+ Hanako（执行）的三Agent流水线不是为这个特定业务定制的——它是一种通用的"感知→规划→执行"范式。任何涉及"接收自然语言指令、在真实系统中执行的"Agent应用，都可以复用这套架构。

## 六、结语

一个好的AI Agent不是能调多少个API的"工具人"，而是：

- 能理解你的意图——哪怕你表达得不完美
- 知道怎么做——包括API文档里没写的部分
- 做错了能自己修——不是终止报错，而是找备用路径
- 修完能记住——下次不会再犯同样的错误
- 总结成规则——让所有跟你协作的Agent都受益

这个淀粉贸易系统就是这样一个Agent的完整实践记录。它不是"AI+业务"的简单叠加，而是**Agent理解业务、执行规则、处理异常、持续学习**这一完整闭环的工程验证。
