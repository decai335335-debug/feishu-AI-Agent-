# 淀粉贸易业务管理系统 · 开发日志

---

## 1. 项目起源

徐付文经营淀粉贸易，日常需要管理70+客户的采购、销售、回款、物流。原本在飞书表格里手动操作，每天录入几十条记录、关联订单、核对回款，效率低且易出错。

需求：用AI Agent替代手工录入，实现"说一句话，自动记一笔账"。

技术约束：飞书API不完善（公式不可写入、DuplexLink格式隐晦）、bash沙盒拦截、浏览器不稳定、IP动态变化。

## 3. 踩坑记录

| 问题 | 根因 | 解决方案 | 涉及版本 |
|------|------|---------|:--:|
| 公式字段复制后丢失 | `property.formula`为空，实际存在`formula_expression` | 读`formula_expression`反向导出 | v2 |
| DuplexLink写入后record_ids空 | 传了对象格式`{table_id:...}` | 纯数组`["record_id"]` | v1-v3 |
| 回款关联到错误客户 | 修正时硬编码order_id不验客户名 | GET双方记录验证客户名一致 | v4-v5 |
| 反向链接被PUT覆盖 | PUT替换语义非追加 | 先GET→拼接→PUT | v5 |
| OAuth码被拆分消耗 | 查ID+创建分两步 | 同evaluate内完成所有操作 | v1-v3 |
| 吨数匹配漏掉1.2吨 | JS浮点`1.2!==1.2000000000000002` | 吨数×100转整数比对 | v2 |
| 浏览器display surface崩溃 | 虚拟显示驱动断开 | restart后重试，抢在断开前evaluate | 全版本 |
| 多出口IP被拦 | token和数据请求走不同IP | 两个IP都加白名单；OAuth用户token绕过 | 全版本 |
| `.json()`报Script failed | 浏览器不支持 | `text()+JSON.parse()` | v1 |
| 送货关联失败5次 | code=0就以为成功，不回读验证 | 逐条GET回读record_ids | v3-v7 |

## 4. 设计决策

**为什么用飞书而非MySQL？**
飞书多维表格自带公式引擎、关联字段（DuplexLink）、权限管理，零运维。适合个人/小团队。MySQL需要服务器、后端接口、前端展示，运维成本高。

**为什么用浏览器evaluate而非Node.js？**
Hanako沙盒拦截了bash和外部二进制。浏览器的fetch+JSON.parse是最小可行方案。代价是每次操作需要OAuth授权+处理CSP限制。

**为什么FIFO拆分选择"删旧建新"而非"在原记录上改金额"？**
实测多次：改金额→忘改关联→忘写反向→覆盖旧值。删旧建新虽然多一步，但每步可验证，不残留脏数据。

**为什么回款要逐单绑定而非多订单链接？**
`SUM([5回款明细💵 2].[回款金额（元）])`对每个关联订单都加全额。一笔100,000挂在3个订单上=每个显示100,000=翻3倍。
